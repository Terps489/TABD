## Целевые переменные

Модель TFT прогнозирует **одновременно** 9 показателей в формате
"за час" — стандартный горизонт прогноза 24 часа вперёд.

### Топливо (8 показателей)

| Код | Название | Что это | Единицы |
|---|---|---|---|
| `total_fuel_sales` | Общие продажи топлива | Сумма продаж всех видов топлива на АЗС | литров/час |
| `sales_AI92` | АИ-92 | Бензин с октановым числом 92 | литров/час |
| `sales_AI95` | АИ-95 | Бензин с октановым числом 95 (наиболее популярный) | литров/час |
| `sales_AI98` | АИ-98 | Бензин премиум-сегмента | литров/час |
| `sales_DT_EURO` | ДТ Евро+ | Дизельное топливо стандарта Евро | литров/час |
| `sales_DT_TANEKO` | ДТ ТАНЕКО | Дизельное топливо ТАНЕКО (бренд Татнефти) | литров/час |
| `sales_DT_SUMMER` | ДТ Летнее | Дизель для тёплого сезона | литров/час |
| `sales_DT_WINTER` | ДТ Зимнее | Дизель для холодного сезона (с присадками) | литров/час |

### Сопутствующие товары (1 показатель)

| Код | Название | Что это | Единицы |
|---|---|---|---|
| `shop_total_revenue` | Выручка магазина | Сумма продаж всех товаров (напитки, закуски, кофе, табак, автотовары) | руб/час |

## Типы прогнозов

TFT выдаёт **квантильный прогноз** — не одно число, а диапазон возможных
значений с разной степенью уверенности.

| Квантиль | Смысл | Когда использовать |
|---|---|---|
| **P10** | С вероятностью 90 % реальное значение будет **выше** | Консервативное планирование запасов (минимум) |
| **Медиана** | Наиболее вероятное значение прогноза | Базовое планирование |
| **P90** | С вероятностью 90 % реальное значение будет **ниже** | Оценка пиковой нагрузки, страховой запас |

**Интервал P10–P90** — 80-% доверительный коридор. Узкий = модель уверена,
широкий = высокая неопределённость (часы пик, необычная погода).

## Вкладки дашборда

### 1. Обзор сети

Агрегированный взгляд на всю сеть из 25 АЗС.

**Фильтры:** период (начало/конец), агрегация (день/неделя/месяц).

**Графики:** KPI-карточки (АЗС / общий объём / средние л/час), динамика
продаж по 7 видам топлива, структура продаж (pie), трафик по типам
транспорта, выручка магазина по категориям, heatmap «час × день недели».

### 2. Анализ АЗС

Глубокая аналитика **одной выбранной АЗС**.

**Фильтры:** конкретная АЗС, виды топлива (галочки).

**Графики:** ежедневные продажи, суточный паттерн (по часам), недельный
паттерн (по дням), влияние погоды (ясно/дождь/снег/…).

### 3. Прогнозы TFT

**Самая важная вкладка — здесь работает модель.**

**Фильтры:** АЗС для прогноза, показатель (один из 9 таргетов),
кнопка **⚙ Настройки**.

**Что на графике:**
- **Слева от вертикальной линии** — фактические значения (синяя линия).
- **Справа от вертикальной линии** — прогноз TFT:
  - Жёлтая пунктирная линия — медиана.
  - Цветная зона — интервал P10–P90.
- **Вертикальная пунктирная линия в `0`** — текущий момент.

**Настройки (⚙):** часы истории (1…168) и часы прогноза (1…720). Дефолты
берутся из `src/config.py` (`FORECAST_CHART_HISTORY_HOURS` и
`FORECAST_CHART_FUTURE_HOURS`), сохранённые значения хранятся в
`localStorage` браузера и переживают перезагрузку страницы. Кнопка
«Сброс» возвращает к дефолтам конфига. При прогнозе **>24 ч** включается
итеративный rollout (медленнее, точность падает с горизонтом — это видно
по расширению P10–P90).

### 4. Прогноз — таблица

Числовой прогноз в табличном виде — для копирования, экспорта, точных
значений.

**Фильтры:** АЗС, показатель, **горизонт** (24 / 48 / 168 / 720 часов),
кнопка «Рассчитать».

**Таблица:** «Ч вперёд / Время / P10 / Медиана / P90», по 24 строки на
страницу, сортируемая. Для горизонта > 24 ч запускается итеративный
rollout: модель шагает по 24 ч, медианы предыдущего шага подаются
в энкодер как «история» следующего.

### 5. Рекомендации

**Автоматические инсайты + факторный анализ.**

**Содержание:**
- **Текстовые рекомендации** — топ-5 факторов с подсказками к действию
  («трафик — наиболее важный; усильте маркетинг в часы пик 7-9, 17-19»).
- **Топ-10 АЗС по продажам** — bar-chart лидеров.
- **Важность факторов** — топ-15 факторов по абсолютной корреляции
  с `total_fuel_sales` (через `feature_importance.json`).
- **Эффект акций** — сравнение средних продаж «с акцией» vs «без акции».
- **Корреляционная матрица 9 таргетов × 12 факторов** — heatmap RdBu
  с центром в 0. Видно, для каких продуктов какой фактор работает
  сильнее и в какую сторону (трафик, погода, час пик, промо, цены
  конкурентов).

### 6. Метрики

**Сравнение TFT с baseline-моделями на одной и той же 24-часовой
валидационной выборке.**

Baselines:

- `naive_yesterday` — ŷ(t) = y(t − 24): прогноз = значение того же часа
  предыдущего дня. Сильный baseline на сильной суточной сезонности.
- `seasonal_naive_week` — ŷ(t) = y(t − 168): прогноз = значение того же
  часа неделю назад. Учитывает недельную сезонность.

**Содержание вкладки:**

- **Macro-avg по моделям** — bar-charts MAE / RMSE / MAPE / SMAPE,
  усреднённые по 9 таргетам.
- **Таблица per-target** — MAE, RMSE, MAPE, SMAPE по каждому из 9 таргетов
  для выбранной модели. Опционально — фильтр по конкретной АЗС.
- **Heatmap SMAPE** — модель × таргет (зелёное лучше).

Артефакты: `outputs/metrics/metrics.csv` (long: model × target × station),
`outputs/metrics/summary.json` (агрегаты).

### 7. Настройки

**Запуск полного pipeline из UI** — для пользователя, который не хочет
открывать терминал.

**Секции:**

1. **Загрузка CSV** — два `dcc.Upload` для `detailed_data.csv` и
   `stations_metadata.csv`. Файлы валидируются как CSV (заголовок читается
   как DataFrame), затем сохраняются в `data/`.
2. **Запуск задачи** — кнопки:
   - «Обучить модель» — `python run.py --mode train` (опция «быстрый
     режим» = `--quick`, 5 АЗС).
   - «Сгенерировать прогнозы» — `python run.py --mode predict`.
   - «Только метрики» — `python run.py --mode evaluate` (быстрый
     пересчёт MAE/RMSE/MAPE/SMAPE + baselines без перегенерации
     прогнозов).
   - «Остановить» — `taskkill` текущего subprocess.
3. **Статус и журнал** — алерт со статусом активной задачи (PID, время
   старта/завершения, код выхода) и `<pre>` с хвостом stdout/stderr,
   обновляется каждые 2 секунды через `dcc.Interval`. После успешного
   завершения вкладка «Метрики» автоматически перечитывает файлы.

**Архитектура:** `src/training_runner.py` запускает `run.py` как
отдельный subprocess (sys.executable + правильный CWD), пишет PID и путь
к лог-файлу в `logs/runner/state.json`. Дашборд остаётся отзывчивым,
параллельный запуск двух задач блокируется на уровне runner-state.

## Графики

Все графики интерактивные:

- **Hover** — точные значения при наведении.
- **Zoom** — выделение прямоугольника мышью.
- **Pan** — потянуть график (после зума).
- **Скрыть линию** — клик на легенду; **изоляция** — двойной клик.
- **Reset** — двойной клик на пустое место графика.
- **Скачать PNG** — иконка камеры в правом верхнем углу графика.

## Файлы прогнозов

После `python run.py --mode predict` в `outputs/forecasts/` появляются
9 CSV-файлов (по одному на таргет). Структура:

```csv
station_id,step,forecast_p10,forecast_median,forecast_p90
0,0,5.06,5.82,11.82
0,1,4.82,5.10,10.82
...
24,23,...
```

- `station_id` — ID станции (0–24).
- `step` — шаг прогноза (0–23, часов вперёд).
- `forecast_*` — три квантиля.

Итого: 25 АЗС × 24 часа = 600 строк на файл. Эти CSV — *training-time
validation*; дашборд читает их для горизонта ≤ 24 ч. Для большего
горизонта запускается on-demand iterative rollout в памяти процесса
дашборда (без сохранения на диск).
